# GameForge 🎮🔨

**AI Game Development Studio** — a multi-agent framework that turns a game design document into a playable game.

## How It Works

```
game_design.md → Producer → execution_plan.json → Translator → runnable agents → playable game
```

**Four layers:**

| Layer | What It Does | You Write? |
|---|---|---|
| **Producer** | Reads game design, outputs execution plan | ✅ Yes |
| **Translator** | Converts plan to multi-agent code (AutoGen/CrewAI/etc.) | ✅ Yes |
| **Dev Loop** | Designer ⇄ Implementer ⇄ Playtester continuous cycle | Generated |
| **Adapter** | Optional headless engine for batch simulation | Built by agents |

## Quick Start

```bash
# Generate an execution plan from a game design
gameforge plan examples/hu/game_design.md -o plan.json

# Review the plan
gameforge show plan.json

# Translate to runnable AutoGen code
gameforge translate plan.json --runtime autogen -o ./generated

# Run the agents
cd generated && python main.py
```

## Project Structure

```
gameforge/
├── producer/           # Layer 1: game_design.md → execution_plan.json
│   └── producer.py     # YOUR CODE
├── translator/         # Layer 2: plan → runnable multi-agent code
│   ├── base.py         # Abstract Translator interface
│   └── autogen_translator.py  # YOUR CODE
├── devloop/            # Layer 3: generated by Translator
├── adapter/            # Layer 4: optional, built by agents
│   └── base.py         # Abstract GameAdapter interface
├── models/             # Shared data models (Pydantic)
│   ├── plan.py         # ExecutionPlan, Milestone, AgentConfig
│   └── feedback.py     # Feedback types and routing
├── tools/              # Utilities
│   └── llm.py          # LLM client abstraction
└── cli.py              # CLI entry point

examples/
└── hu/                 # Example: HU roguelike mahjong
    └── game_design.md
```
